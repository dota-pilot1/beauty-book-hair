# BeautyBook 프로젝트 컨셉 설명

> "이 프로젝트는 단순한 미용실 예약 데모가 아니라,
> 보일러플레이트가 실제 운영 자산으로 자라날 수 있는지를 검증하는 실험이다."

---

## 1. 왜 만들었나

회사에서 신규 프로젝트나 리뉴얼이 떨어지면, 매번 비슷한 과정을 다시 밟습니다.

- 인증·권한·메뉴·관리자 화면은 매번 새로 짠다.
- 디자인 토큰, 공통 폼, 테이블, 모달도 매번 다시 깐다.
- "지난 프로젝트에서 잘 만들어둔 그거"는 정작 다음 프로젝트로 잘 넘어오지 않는다.

이 반복을 줄이기 위해 보일러플레이트([hibot-docu.com](https://hibot-docu.com))를 만들었지만,
보일러플레이트만 두면 **"빈 골격"** 에 머물기 쉽습니다. 실제 운영 환경에서 깎이고
확장돼 봐야, 그때 비로소 어떤 부분이 자산이 될 수 있는지 드러납니다.

**BeautyBook**은 그 검증을 위한 **첫 번째 실전 파일럿**입니다.

---

## 2. 3단 구조로 본 이 프로젝트의 위치

```
[ Boilerplate ]   →   [ Feature 자산 ]   →   [ Pilot ]
   hibot-docu          예약 / 권한 /          BeautyBook
   (시작용 골격)        메뉴 / 게시판          (미용실 운영)
                       (재사용 단위)          (검증 + 레퍼런스)
```

| 층위 | 역할 | 이 프로젝트에서의 예 |
|------|------|---------------------|
| **Boilerplate** | 어느 도메인이든 공통으로 필요한 토대 | JWT 인증, RBAC, 공통 에러 처리, AdminShell, i18n, 테마 스위처 |
| **Feature 자산** | 도메인을 가지지만 다른 프로젝트로 떼어갈 수 있는 단위 | 예약 슬롯 시스템, 직원·근무시간 관리, 게시판 + 헤더 메뉴 자동 연동, Lexical 에디터 |
| **Pilot** | 위 둘이 실제로 운영되는 모습 | dxline-tallent.com 미용실 예약 운영 |

> **핵심 가설**: 진짜 자산은 보일러플레이트도 파일럿도 아닌 **가운데(Feature) 층**이다.
> 프로젝트 단위로 묶어두면 재사용이 안 되고, 기능 단위로 잘라내야 다음 프로젝트가 빠르다.

---

## 3. 이 프로젝트가 검증하려는 것

### 3-1. 보일러플레이트가 "확장에 견디는가"
- 인증·권한·공통 컴포넌트가, 미용실 도메인 기능을 얹어도 깨지지 않는가
- 새 도메인 모듈(예: 게시판)을 추가할 때, 보일러플레이트가 길을 알려주는가 막는가

### 3-2. 어떤 단위가 "다음 프로젝트로 떼어갈 수 있는가"
실제로 떼어가 본다는 가정으로 만들고, 떼어갈 수 있게 다듬습니다.

| 후보 자산 | 떼어가기 쉬운 정도 | 메모 |
|----------|------------------|------|
| 게시판 + 헤더 메뉴 자동 연동 | ★★★★★ | 도메인 의존이 거의 없음. 콜센터·사내 포털에 그대로 이식 가능 |
| 예약 슬롯(30분 단위, 겹침/근무시간/휴무 체크) | ★★★★ | 미용실/병원/스터디룸/렌탈 도메인 공통 |
| RBAC + Role↔Permission 관리 UI | ★★★★★ | 어느 어드민에든 그대로 이식 |
| Lexical 에디터(이미지 업로드·리사이즈·정렬) | ★★★★ | 게시판/공지/CMS 어디든 |
| AdminShell(사이드바 모핑, 일자바 모드) | ★★★ | 시각 컨셉이 묶여 있어 토큰화 필요 |
| 테마 스위처 + i18n | ★★★★★ | 100% 보일러플레이트로 흡수 가능 |

### 3-3. 운영해 봐야만 보이는 것들
- 운영 배포 후 영업시간(SUNDAY) 체크처럼, **데모로는 절대 안 잡히는 운영 이슈**가 어디서 터지는가
- CloudFront/S3에서 같은 버킷의 이미지 폴더를 보호해야 하는 것 같은,
  배포 스크립트에 박혀야만 다음 프로젝트가 안 깨지는 지식은 무엇인가
- 이런 운영 지식은 결국 `CLAUDE.md` / `배포 가이드/` 같은 **문서 자산**으로 보일러플레이트로 흘러가야 한다

---

## 4. 자산화 사이클

```
  파일럿(BeautyBook)에서 운영
        │
        ▼
  반복되는 패턴 발견
        │
        ▼
  Feature 단위로 추출 (게시판, 예약, RBAC …)
        │
        ▼
  hibot 보일러플레이트로 역류 + 다음 파일럿이 더 빠르게 시작
```

이 사이클이 한 바퀴 돌 때마다 보일러플레이트는 한 단계 단단해지고,
다음 프로젝트는 그 만큼 "처음부터 다시 짜는 일"이 줄어듭니다.

## 5. 곁들여 — 같은 결론에 도달한 한국·해외 사례

이 컨셉이 우리 팀만의 발상은 아닙니다. 이미 같은 답에 도달한 사례들이 있습니다.

### 한국 회사들이 푸는 방식

- **토스 — Frontend Fundamentals**: "좋은 코드 = 변경하기 쉬운 코드"를 사내 표준 지침서로 만들어 운영. → https://frontend-fundamentals.com/
- **토스 — 사내 라이브러리(`frontend-libraries`)**: 프로젝트 안에서 만들고, 패턴이 보이면 라이브러리로 승격하는 흐름. → https://toss.tech/article/toss-frontend-chapter
- **토스 — `@use-funnel` 개발기**: "퍼널 UI"라는 한 피처를 라이브러리로 떼어낸 대표 사례. 이 프로젝트의 "예약 슬롯"·"게시판"이 같은 길을 지향. → https://toss.tech/article/use-funnel-1
- **카카오스타일 — 디자인 시스템 재구축기**: 토큰을 모듈화해 공통화. → https://devblog.kakaostyle.com/ko/2024-12-13-1-rebuilding-frontend-design-system/
- **우아한형제들 — 파일럿 프로젝트 제도**: 신입이 보일러플레이트를 익히고 작은 피처 하나를 끝까지 만들어보는 사내 온보딩. **"파일럿"이라는 이 프로젝트의 명명과 직접 맞닿음**. → https://techblog.woowahan.com/2592/
- **카카오 — 어드민 UI 컴포넌트 모노레포**: "전사 공통"이 아니라 **"어드민 도메인 공통"** 이라는 중간 레이어. 우리도 같은 발상이 가능. → https://tech.kakao.com/posts/413
- **FEConf 2025 — 모노레포 절망편**: 4년간 운영한 모노레포 1개 → 14개 레포로 분리. **"공통화 = 정답"이 아니라는 반면교사**. → https://speakerdeck.com/mmmaxkim/feconf-2025-monorepo-jeolmangpyeon-14gae-reporo-buhwalhagiggaji-geolrin-1nyeon
- **INFCON 2024 — 디자인 시스템의 명과 암 (원티드)**: 공통화의 부작용("커스터마이징이 안 돼서 결국 안 씀") 솔직한 회고. → https://www.inflearn.com/conf/infcon-2024/session-detail/905/

## 6. 한 줄 요약

> **BeautyBook은 미용실 예약 서비스이자, hibot 보일러플레이트가 실전에서 자산화되는 과정을 검증하는 첫 번째 파일럿이다.**
