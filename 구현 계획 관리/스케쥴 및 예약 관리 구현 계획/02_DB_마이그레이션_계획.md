# 02. DB 마이그레이션 계획

## 목표

예약 가능 시간 계산과 예약 관리를 위한 최소 테이블을 만든다.

## 테이블 1: business_hours

매장 기본 영업시간.

```text
id
day_of_week
open_time
close_time
closed
created_at
updated_at
```

권장 제약:

- day_of_week unique
- closed가 true이면 open_time, close_time은 nullable 허용

## 테이블 2: designers

예약 가능한 디자이너.

```text
id
name
profile_image_url
introduction
active
display_order
created_at
updated_at
```

권장 제약:

- active가 false이면 고객 예약 화면에 노출하지 않는다.
- display_order로 고객 화면 정렬을 제어한다.

## 테이블 3: designer_services

디자이너가 수행 가능한 시술 매핑.

```text
id
designer_id
beauty_service_id
active
created_at
updated_at
```

권장 제약:

- designer_id + beauty_service_id unique
- active가 false이면 해당 시술 선택 시 디자이너 후보에서 제외한다.

## 테이블 4: designer_working_hours

디자이너별 기본 근무시간.

```text
id
designer_id
day_of_week
start_time
end_time
working
created_at
updated_at
```

권장 제약:

- designer_id + day_of_week unique
- working이 false이면 start_time, end_time nullable 허용

## 테이블 5: blocked_times

예약 불가 시간.

```text
id
designer_id nullable
start_at
end_at
reason
block_type
created_at
updated_at
```

block_type 예:

- STORE_CLOSED
- DESIGNER_OFF
- LUNCH
- EDUCATION
- PERSONAL
- ETC

## 테이블 6: reservations

예약 데이터.

```text
id
customer_id nullable
customer_name
customer_phone
designer_id
beauty_service_id
start_at
end_at
status
customer_memo
admin_memo
created_at
updated_at
```

권장 인덱스:

- designer_id + start_at
- designer_id + end_at
- status
- customer_phone

## 중복 예약 방지

DB unique index만으로 시간 구간 겹침을 완벽히 막기는 어렵다.

MVP에서는 서비스 레이어에서 아래 조건으로 겹침을 검사한다.

```text
existing.startAt < newEndAt
AND existing.endAt > newStartAt
AND existing.status IN (REQUESTED, CONFIRMED)
```

동시성은 이후 단계에서 pessimistic lock 또는 transactional 재검증으로 보강한다.
