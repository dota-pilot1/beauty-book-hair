# 스케쥴 및 예약 관리 전체 구현 로드맵

## 결론

스케쥴 관리와 예약 관리는 **같이 설계하고, 구현은 단계별로 나눠서** 가는 것이 맞다.

이유는 단순하다.

- 예약 가능한 시간은 스케쥴 데이터 없이는 계산할 수 없다.
- 스케쥴만 만들고 예약을 늦게 붙이면 `이미 예약된 시간 제외` 로직을 다시 뜯게 된다.
- 예약 관리부터 만들면 가능한 시간 기준이 없어서 고객 예약 화면이 계속 하드코딩에 머문다.

따라서 구현 순서는 아래처럼 잡는다.

1. 스케쥴/예약 도메인 모델 설계
2. 영업시간/근무시간/차단시간 관리
3. 예약 엔티티와 상태 모델 추가
4. 시술 가능 디자이너 조회 구현
5. 예약 가능 슬롯 계산 API 구현
6. 고객 예약 플로우 연동
7. 관리자 예약 목록/상태 변경 구현
8. 충돌 방지와 검증 보강
9. 캘린더/운영 화면 개선

## 핵심 원칙

- 스케쥴은 마스터 데이터다.
- 예약은 트랜잭션 데이터다.
- 예약 가능 시간은 저장하지 않고 우선 계산한다.
- 예약 확정 시점에는 반드시 서버에서 다시 충돌 검증한다.
- 프론트의 슬롯 선택은 UX일 뿐이고, 최종 판단은 백엔드가 한다.

## 최소 MVP 범위

MVP에서는 아래까지만 제대로 만들면 된다.

- 매장 요일별 영업시간
- 디자이너 요일별 근무시간
- 디자이너별 가능한 시술
- 특정 날짜/시간 예약 불가 설정
- 예약 생성
- 예약 상태 관리
- 예약 가능 시간 조회
- 시술 + 날짜 기준 시간별 가능한 디자이너 조회
- 고객 예약 화면에서 실제 슬롯 선택
- 관리자 예약 목록에서 승인/확정/취소

## 먼저 구현해야 하는 것

1차 구현은 `디자이너-시술 매핑 + 스케쥴 기본 관리 + 예약 엔티티 + 슬롯 계산`까지 같이 가야 한다.

관리자 예약 목록 UI까지 한 번에 완성하지 않아도 되지만, 예약 가능 슬롯 계산에서 기존 예약을 제외하려면 예약 테이블은 초기에 들어가야 한다.

## 피해야 할 방향

- 예약 시간을 문자열로 저장하지 않는다.
- 고객 화면의 날짜/시간 슬롯을 하드코딩하지 않는다.
- 프론트에서만 예약 중복을 막지 않는다.
- 디자이너별 근무/예약/차단을 무시하고 매장 전체 슬롯만 계산하는 구조로 고정하지 않는다.
- 고객 화면은 시술→디자이너→날짜/시간 3단계로 확정. 시술 기준 가능한 디자이너 먼저 선택 후 해당 디자이너의 슬롯을 보여준다.
- 예약 상태를 단순 boolean으로 두지 않는다.
