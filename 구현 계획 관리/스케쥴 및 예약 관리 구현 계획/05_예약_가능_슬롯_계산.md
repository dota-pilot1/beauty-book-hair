# 05. 예약 슬롯 계산

## 목표

고객 화면에서 시술을 선택하면 날짜별 전체 시간대를 보여주고, 각 시간대의 예약 가능 여부와 해당 시간에 가능한 디자이너 목록을 함께 내려주는 API를 만든다.

가능한 슬롯만 내려주면 고객은 어떤 시간이 이미 찼는지, 어떤 시간이 매장 차단 시간인지 이해하기 어렵다. MVP에서는 30분 단위로 후보 시작 시간을 만들고, 시술 소요시간과 겹침 여부에 따라 `AVAILABLE`, `REQUESTED`, `RESERVED`, `BLOCKED` 상태를 내려준다.

이 단계에서는 디자이너를 먼저 고르게 강제하지 않는다. 기본 조회는 `beautyServiceId + date` 기준으로 가능한 시간 전체를 계산하고, 필요하면 `designerId`를 optional 필터로 받아 특정 디자이너 기준으로 좁힌다.

## 입력

```text
beautyServiceId
date
designerId optional
```

`designerId`가 있으면 선택한 시술을 수행할 수 있는 디자이너여야 한다. 슬롯 계산 전 서버에서 `designerId + beautyServiceId` 가능 여부를 검증한다.

`designerId`가 없으면 해당 시술을 수행 가능한 active 디자이너 전체를 대상으로 계산한다. 같은 시작 시간에 가능한 디자이너가 1명 이상이면 그 시간 카드는 선택 가능하다.

## 출력

```json
[
  {
    "slotId": "2026-04-29T10:30:00",
    "startAt": "2026-04-29T10:30:00",
    "endAt": "2026-04-29T11:30:00",
    "durationMinutes": 60,
    "unitMinutes": 30,
    "occupiedUnitCount": 2,
    "status": "AVAILABLE",
    "selectable": true,
    "availableDesigners": [
      {
        "id": 1,
        "name": "하린"
      },
      {
        "id": 2,
        "name": "민서"
      }
    ],
    "reason": "예약 요청 가능"
  },
  {
    "slotId": "2026-04-29T16:30:00",
    "startAt": "2026-04-29T16:30:00",
    "endAt": "2026-04-29T17:30:00",
    "durationMinutes": 60,
    "unitMinutes": 30,
    "occupiedUnitCount": 2,
    "status": "RESERVED",
    "selectable": false,
    "availableDesigners": [],
    "reason": "가능한 디자이너가 없습니다."
  }
]
```

확장 조회가 필요하면 여러 날짜를 한 번에 내려줄 수 있다.

```http
GET /api/reservation-slot-days?beautyServiceId=10&fromDate=2026-04-29&days=14
```

응답은 날짜별 그룹을 기준으로 둔다.

```json
[
  {
    "date": "2026-04-29",
    "slots": []
  }
]
```

## 계산 공식

```text
디자이너별 예약 가능 상태 =
매장 영업시간
∩ 해당 디자이너 근무시간
- 매장 전체 차단 시간
- 해당 디자이너 차단 시간
- 해당 디자이너의 기존 예약 시간

시간 카드 상태 =
같은 startAt/endAt 구간에서 가능한 디자이너 목록을 집계한 결과
```

## 계산 단계

1. date의 요일을 구한다.
2. 선택한 시술을 수행 가능한 active 디자이너 목록을 조회한다.
3. `designerId` 필터가 있으면 해당 디자이너만 남기고 수행 가능 여부를 검증한다.
4. BusinessHour를 조회한다.
5. 디자이너별 DesignerWorkingHour를 조회한다.
6. 매장이 closed거나 디자이너가 working false면 해당 디자이너는 가능 후보에서 제외한다.
7. 영업시간과 디자이너 근무시간의 교집합을 구한다.
8. 해당 날짜의 BlockedTime을 조회한다.
9. 해당 날짜의 Reservation 중 REQUESTED/CONFIRMED 상태를 조회한다.
10. slotUnitMinutes 단위로 후보 시작 시간을 만든다.
11. 후보 시작 시간 + 시술 소요시간으로 endAt을 계산한다.
12. 디자이너별로 근무 가능 구간, 차단 시간, 기존 예약과 겹치는지 판단한다.
13. 같은 후보 시작 시간에 가능한 디자이너를 `availableDesigners`로 집계한다.
14. 가능한 디자이너가 1명 이상이면 AVAILABLE로 표시한다.
15. 아무 디자이너도 가능하지 않으면 가장 강한 차단 사유 기준으로 REQUESTED/RESERVED/BLOCKED를 표시한다.

## 기본 정책

- slotUnitMinutes: 30분
- 30분 단위는 미용실 예약에서 과하지 않고 일반적인 기본값으로 둔다.
- 슬롯 간격은 30분이지만, 실제 점유 시간은 시술의 durationMinutes를 따른다.
- `occupiedUnitCount = ceil(durationMinutes / slotUnitMinutes)`로 내려 프론트가 선택 범위를 시각화할 수 있게 한다.
- REQUESTED 예약도 점유로 처리
- CANCELLED 계열은 점유하지 않음
- 현재 시간 이전 슬롯은 BLOCKED 또는 응답 제외 중 선택할 수 있으나, 고객 화면 일관성을 위해 MVP에서는 BLOCKED가 낫다.
- 당일 예약 마감 시간 정책은 2차로 미룬다.
- 프론트는 `occupiedUnitCount`를 하이라이트 용도로만 사용하고, 선택 가능 여부의 최종 판단은 서버 응답을 따른다.
- 예약 생성 시 서버에서 동일한 조건으로 다시 검증한다.

## 슬롯 상태

- AVAILABLE: 예약 요청 가능
- REQUESTED: 다른 고객의 승인 대기 예약과 겹침
- RESERVED: 확정 예약과 겹침
- BLOCKED: 영업시간 외, 근무시간 외, 오픈 준비, 휴무, 차단 시간

프론트에서는 AVAILABLE만 클릭 가능하게 두고 나머지는 disabled로 표시한다. AVAILABLE 카드 선택 시 선택된 시작 시간부터 `occupiedUnitCount`만큼 연결된 블록처럼 즉시 강조한다.

## 겹침 판정

```text
a.start < b.end AND a.end > b.start
```

경계가 딱 붙는 경우는 겹치지 않는 것으로 본다.

예:

- 기존 예약 10:00-11:00
- 새 예약 11:00-12:00
- 가능

## 완료 기준

- 고객 예약 화면의 시간 슬롯 하드코딩을 제거할 수 있다.
- 전체 시간대가 30분 단위 후보로 보인다.
- 디자이너를 먼저 선택하지 않아도 시술 + 날짜 기준으로 가능한 시간과 가능한 디자이너가 보인다.
- 시간 카드마다 가능한 디자이너 이름이 보인다.
- 시간 카드 선택 시 시술 소요시간만큼 차지하는 범위가 즉시 표시된다.
- 이미 예약된 시간은 RESERVED로 내려온다.
- 승인 대기 요청과 겹치는 시간은 REQUESTED로 내려온다.
- 차단 시간이 있으면 해당 구간은 BLOCKED로 내려온다.
- 프론트는 AVAILABLE만 선택 가능하게 처리한다.
