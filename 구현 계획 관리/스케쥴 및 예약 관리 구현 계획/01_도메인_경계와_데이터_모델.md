# 01. 도메인 경계와 데이터 모델

## 목표

스케쥴과 예약을 한 번에 설계하되, 코드 구조는 분리한다.

## 도메인 구분

### schedule

예약 가능 여부를 판단하기 위한 기준 데이터.

- 매장 영업시간
- 디자이너 근무시간
- 예약 불가 시간
- 임시 휴무
- 슬롯 계산 정책

### reservation

고객이 실제로 요청하거나 확정한 예약 데이터.

- 예약 요청
- 예약 확정
- 예약 취소
- 노쇼
- 관리자 메모
- 고객 요청사항

### designer

디자이너와 시술 가능 여부를 관리하는 기준 데이터.

- 디자이너 프로필
- 디자이너별 가능한 시술
- 디자이너별 노출 여부

## 핵심 엔티티

### BusinessHour

매장 요일별 영업시간.

- dayOfWeek
- openTime
- closeTime
- closed

### DesignerWorkingHour

디자이너 요일별 근무시간.

- designerId
- dayOfWeek
- startTime
- endTime
- working

### Designer

예약 가능한 디자이너.

- name
- profileImageUrl
- introduction
- active
- displayOrder

초기에는 사용자 계정과 분리해도 된다. 운영자/직원 로그인과 강하게 묶기 전에 예약 가능한 담당자 데이터로 먼저 두는 편이 구현이 단순하다.

### DesignerService

디자이너가 수행 가능한 시술 매핑.

- designerId
- beautyServiceId
- active

고객 예약 플로우에서는 시술을 먼저 선택하고, 이 매핑을 기준으로 시간별 가능한 디자이너를 계산한다. 디자이너 선택은 필수 선행 단계가 아니라 시간 선택 후 확정하거나 optional 필터로 사용할 수 있다.

### BlockedTime

특정 날짜/시간에 예약을 막는 데이터.

- designerId nullable
- startAt
- endAt
- reason
- blockType

`designerId`가 null이면 매장 전체 차단으로 본다.

### Reservation

실제 예약 데이터.

- customerId nullable
- customerName
- customerPhone
- designerId
- beautyServiceId
- startAt
- endAt
- status
- customerMemo
- adminMemo

## 예약 상태

초기 상태는 아래 정도면 충분하다.

- REQUESTED: 고객 예약 요청
- CONFIRMED: 관리자 확정
- CANCELLED_BY_CUSTOMER: 고객 취소
- CANCELLED_BY_ADMIN: 관리자 취소
- COMPLETED: 시술 완료
- NO_SHOW: 노쇼

## 주요 정책

- 고객 예약 플로우는 기본적으로 `시술 선택 -> 날짜/시간 선택 -> 가능한 디자이너 선택 또는 자동 선택` 순서로 둔다.
- 디자이너 목록은 선택한 시술과 선택한 시간에 실제로 가능한 디자이너만 노출한다.
- 예약 슬롯은 기본적으로 `beautyServiceId + date` 조합으로 계산하고, `designerId`는 optional 필터로 둔다.
- 각 시간 슬롯은 해당 시간에 가능한 디자이너 목록을 함께 내려준다.
- REQUESTED를 시간 점유로 볼지 결정해야 한다.
- MVP에서는 REQUESTED도 점유로 본다.
- 그래야 같은 시간에 여러 고객이 요청하는 상황을 줄일 수 있다.
- 취소 상태는 슬롯 계산에서 제외한다.
