# 시술 정보 관리 구현 계획

## 목표

- 시술 정보를 예약 화면 하드코딩이 아니라 DB 마스터 데이터로 관리한다.
- 시술 카테고리도 별도 마스터 테이블로 분리한다.
- 고객 예약 화면은 `시술 카테고리 -> 시술 선택` 구조로 확장 가능하게 만든다.
- 기존 프로젝트의 DDD 4-Layer 구조를 유지한 채 `beauty_service` 컨텍스트를 확장한다.

## 왜 지금 갈아엎어야 하는가

- 현재 시술 데이터는 전부 가데이터라 운영 기준이 아니다.
- `CUT`, `COLOR` 같은 enum 카테고리만으로는 운영 확장이 어렵다.
- 시술 수가 많아지면 카테고리명 변경, 노출 여부, 정렬 순서, 하위 확장 요구가 바로 생긴다.
- 예약 시간 계산과 예약 저장도 결국 `시술 ID` 기준으로 가야 안정적이다.

## 큰 방향

- 관리자 화면에서 `시술 카테고리` 와 `시술` 을 같이 관리
- 시술은 `category_id` 로 카테고리를 참조
- 고객 예약 화면은 카테고리 기준으로 시술을 조회/필터링
- 예약 엔티티는 나중에 `beauty_service_id` 를 저장

## DDD 구조 기준

컨텍스트 이름은 그대로 `beauty_service` 를 유지한다.

## 컬럼 메모

### beauty_service_categories

- `code`
  - 내부 관리용 고유 코드
  - 예: `CUT`, `PERM`, `COLOR`, `CLINIC`
- `name`
  - 화면 노출용 이름
  - 예: `컷`, `펌`, `염색`, `클리닉`
- `visible`
  - 고객 예약 화면 필터/탭 노출 여부
- `display_order`
  - 카테고리 정렬 제어

### beauty_services

- `code`
  - 내부 관리용 고유 코드
  - 예: `LAYERED_CUT`, `ROOT_COLOR`
- `name`
  - 화면 노출용 이름
- `category_id`
  - 시술 카테고리 FK
- `duration_minutes`
  - 예약 가능 시간 계산의 핵심
- `price`
  - 초기에는 정가 1개만 관리
- `target_gender`
  - 초기에는 `ALL`, `WOMEN`, `MEN`
- `visible`
  - 고객 화면 노출 여부
- `display_order`
  - 관리자 정렬 제어

## 백엔드 구현 순서

1. `beauty_service_categories` 엔티티/리포지토리/서비스/API 추가
2. `beauty_services` 가 `category_id` 를 참조하도록 리팩토링
3. request/response DTO 를 `categoryId` 기준으로 변경
4. 시드 데이터를 `카테고리 -> 시술` 순서로 분리
5. `ErrorCode` 에 카테고리 관련 코드 추가
6. 기존 예약 화면/관리 화면 API 를 category 기반으로 변경

## 프론트 구현 순서

1. `entities/beauty-service-category` 타입/API 추가
2. `entities/beauty-service` 타입을 `category object` 기준으로 변경
3. 관리자 `/beauty-services` 화면에 카테고리 관리 영역 추가
4. 시술 등록 다이얼로그는 카테고리 목록을 조회해서 선택하도록 변경
5. 고객 `/booking` 은 시술 조회 결과의 카테고리 정보를 같이 사용

## 고객 예약 UI 반영 방향

- 현재 하드코딩된 카드 3개를 DB 조회 결과 기반 목록으로 교체
- 시술 수가 많아질 수 있으므로 단순 카드 3개 고정 구조는 지양
- 카테고리 기준 탭/필터 확장이 가능해야 함

추천 구조:

- 카테고리 탭 또는 필터
- 시술 목록
- 선택된 시술 요약

## target_gender 처리 기준

초기에는 아래 정도로 단순화한다.

- `ALL`
- `WOMEN`
- `MEN`

주의:

- 성별을 예약 첫 단계로 강제하지 않는다.
- 우선 시술 속성으로만 두고, 나중에 필터 여부를 운영 기준으로 결정한다.

## 예약 기능과의 연결 포인트

- 예약 생성 시 `beauty_service_id` 저장
- 시술의 `duration_minutes` 기준으로 예약 슬롯 계산
- 시술의 `price` 기준으로 예상 금액 표시
- 시술 목록은 `beauty_service_category` 기준으로 그룹화 또는 필터링 가능

## 결론

- 시술 정보 관리는 예약 기능보다 먼저 들어가야 하는 선행 마스터 데이터이다.
- enum 카테고리로는 운영 확장이 어려우므로 `시술 카테고리 마스터 + 시술 마스터` 2단 구조로 가는 것이 맞다.
- 우선은 `카테고리 CRUD + 시술 CRUD + 고객 예약 화면 조회 연동` 수준으로 시작하고, 이후 예약 계산 로직과 연결한다.
