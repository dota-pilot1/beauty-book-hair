# 주요 테이블 요약

## 전체 구조 요약

현재 프로젝트 DB는 크게 아래 4개 묶음으로 보면 됨.

1. 인증: `users`, `refresh_tokens`
2. 권한: `roles`, `permissions`, `permission_categories`, `role_permissions`
3. 메뉴: `menus`
4. 사이트 설정: `site_settings`

## 테이블별 메모

### `users`

- 용도: 로그인 사용자 계정 저장
- 연관: `roles` 와 N:1
- 핵심 컬럼
- `id`
- `email` unique
- `password_hash`
- `username`
- `role_id`
- `active`
- `created_at`
- `updated_at`
- 메모
- 계정 1개는 반드시 1개의 역할을 가짐
- 현재 고객/직원/관리자 모두 필요하면 이 테이블 기반으로 확장할 가능성이 큼

### `roles`

- 용도: 역할 저장
- 연관
- `users` 와 1:N
- `permissions` 와 N:M
- 핵심 컬럼
- `id`
- `code` unique
- `name`
- `description`
- `system_role`
- `created_at`
- `updated_at`
- 메모
- 기본 시드: `ROLE_USER`, `ROLE_MANAGER`, `ROLE_ADMIN`
- 시스템 기본 역할 여부를 `system_role` 로 구분

### `permissions`

- 용도: 세부 권한 저장
- 연관
- `permission_categories` 와 N:1
- `roles` 와 N:M
- 핵심 컬럼
- `id`
- `code` unique
- `name`
- `description`
- `category_id`
- `created_at`
- `updated_at`
- 메모
- 화면 접근, 수정, 삭제 같은 세부 권한 단위

### `permission_categories`

- 용도: 권한 그룹 분류
- 연관: `permissions` 와 1:N
- 핵심 컬럼
- `id`
- `code` unique
- `name`
- `description`
- `display_order`
- `created_at`
- `updated_at`
- 메모
- 현재 권한 UI 에서 묶음 단위로 보여주기 위한 성격이 강함

### `role_permissions`

- 용도: 역할과 권한 매핑
- 연관
- `roles` 와 N:M 연결용 조인 테이블
- `permissions` 와 N:M 연결용 조인 테이블
- 핵심 컬럼
- `role_id`
- `permission_id`
- 메모
- 엔티티 파일은 없고 `Role.permissions` 의 `@JoinTable` 로 생성되는 구조
- 역할별 체크박스 권한 관리의 실제 저장 지점

### `refresh_tokens`

- 용도: 리프레시 토큰 저장
- 핵심 컬럼
- `id`
- `user_id` unique
- `token`
- `expires_at`
- `created_at`
- 메모
- 현재 구조상 사용자당 1개 리프레시 토큰을 유지하는 방식
- `user_id` 는 FK 객체 연관이 아니라 숫자값으로 보관

### `menus`

- 용도: 헤더 메뉴 및 관리자 메뉴 저장
- 연관: 자기 자신과 N:1, 즉 트리 구조
- 핵심 컬럼
- `id`
- `code` unique
- `parent_id`
- `label`
- `label_key`
- `path`
- `icon`
- `is_external`
- `required_role`
- `required_permission`
- `visible`
- `display_order`
- `created_at`
- `updated_at`
- 메모
- DB 기반 메뉴 관리의 핵심 테이블
- `parent_id` 로 상위 메뉴를 연결
- 현재 메뉴 표시 제어는 `required_role`, `required_permission`, `visible` 조합으로 처리

### `site_settings`

- 용도: 메인/소개 영역 같은 사이트 전역 설정 저장
- 핵심 컬럼
- `id`
- `hero_image_url`
- `intro_title`
- `intro_subtitle`
- `created_at`
- `updated_at`
- 메모
- 사실상 singleton 테이블
- 코드에서 `id = 1` 고정 사용

## 현재 기준 관계도

```text
roles 1 --- N users
roles N --- M permissions   (via role_permissions)
permission_categories 1 --- N permissions
menus 1 --- N menus         (self parent-child)
users 1 --- 1 refresh_tokens (현재 구현 의도상)
site_settings = 단일 행 테이블
```

## 예약 기능 들어갈 때 예상 확장 포인트

현재 없는 테이블이라 앞으로 추가 가능성이 높은 후보:

- `reservations`
- `customers`
- `staffs` 또는 `employees`
- `services`
- `staff_schedules`
- `business_hours`
- `reservation_status_histories`

## 작업 전에 보면 좋은 체크포인트

- 새 엔티티 추가 시 `users` 를 공통 계정 테이블로 재사용할지 먼저 결정
- 예약 도메인에서 역할과 권한을 어디까지 세분화할지 결정
- 직원과 고객을 `users` 로 통합할지 별도 테이블로 분리할지 먼저 결정
- 예약 가능 시간 계산 로직이 들어가면 단순 CRUD 테이블만으로 끝나지 않음
