# 소프트 딜리트 구현 + 배포 시 S3 이미지 소실 (2026-04-25)

## 이슈 1 — `GET /api/reservations/deleted` 500 에러

### 증상
- 프론트에서 삭제 이력 탭을 열면 서버 로그에 반복적으로 500 출력
  ```
  NoResourceFoundException: No static resource api/reservations/deleted
    for request '/api/reservations/deleted'
  ```
- `?date=` 파라미터를 붙이면 동일 에러

### 원인
백엔드 `ReservationController`에 `/deleted` 엔드포인트 자체가 없었음.  
소프트 딜리트 기능은 프론트와 DB 타입 정의에만 부분 구현된 상태였고, Spring Boot 컨트롤러·서비스·리포지토리는 미구현이었음.  
Spring이 매핑을 못 찾으면 정적 리소스 경로로 fallback → `NoResourceFoundException` 발생.

### 구현한 내용

#### `Reservation.java` (엔티티)
```java
@SQLRestriction("deleted_at IS NULL")   // 일반 조회에서 자동 제외
@Column
private Instant deletedAt;

public void softDelete() {
    this.deletedAt = Instant.now();
}
```

#### `ReservationRepository.java`
```java
// @SQLRestriction 우회 — deleted_at IS NOT NULL 인 것만 조회
@Query("SELECT r FROM Reservation r WHERE r.deletedAt IS NOT NULL ORDER BY r.deletedAt DESC")
List<Reservation> findAllDeleted();

@Query("SELECT r FROM Reservation r WHERE r.deletedAt IS NOT NULL AND r.startAt >= :from AND r.startAt < :to ORDER BY r.deletedAt DESC")
List<Reservation> findDeletedByDateRange(@Param("from") Instant from, @Param("to") Instant to);

@Query("SELECT r FROM Reservation r WHERE r.id = :id AND r.deletedAt IS NULL")
Optional<Reservation> findActiveById(@Param("id") Long id);
```

#### `ReservationController.java`
```java
// 소프트 딜리트
@DeleteMapping("/{id}")
@ResponseStatus(HttpStatus.NO_CONTENT)
public void delete(@PathVariable Long id, @AuthenticationPrincipal UserPrincipal principal) {
    reservationService.softDelete(id, principal);
}

// 삭제 이력 조회 — date 없으면 전체, date 있으면 해당 날짜
@GetMapping("/deleted")
public List<ReservationResponse> listDeleted(
        @RequestParam(required = false) @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) LocalDate date,
        @AuthenticationPrincipal UserPrincipal principal) {
    boolean isAdmin = ...;
    if (!isAdmin) throw new BusinessException(ErrorCode.FORBIDDEN);
    List<Reservation> results = date != null
            ? reservationService.findDeletedByDate(date)
            : reservationService.findAllDeleted();
    return results.stream().map(ReservationResponse::from).toList();
}
```

#### `ReservationResponse.java`
`deletedAt` 필드 추가 — 프론트 타입과 동기화.

### 핵심 교훈
`@SQLRestriction("deleted_at IS NULL")`을 엔티티에 달면 모든 Spring Data 쿼리가 자동으로 필터링됨.  
삭제 이력을 조회하려면 반드시 `@Query`로 JPQL을 직접 작성해야 함 (메서드 이름 쿼리는 제한 무시 불가).

## 이슈 2 — 프론트 빌드 TypeScript 에러 (AdminShell compact prop)

### 증상
```
Type error: Type '{ compact: true; description?: undefined; }' is not assignable to type 'CustomerShellProps'.
  Types of property 'description' are incompatible.
    Type 'undefined' is not assignable to type 'string'.
```

### 원인
```tsx
const Shell = isAdmin ? AdminShell : CustomerShell;
const shellProps = isAdmin
  ? { title: "예약 현황", compact: true as const }  // description 없음
  : { title: "예약 현황", description: "..." };

<Shell {...shellProps}>   // TS가 union 양쪽 타입 모두 검사
```
TypeScript는 `Shell`이 `AdminShell | CustomerShell` union이라 `shellProps`가 양쪽 props 타입에 모두 호환되는지 검사.  
admin용 props에 `description`이 없어서 `CustomerShellProps` 검사 실패.

### 수정
Shell을 공유하지 않고 조건부로 분리:
```tsx
const innerContent = (
  <div className="space-y-4">
    {/* 공통 JSX */}
  </div>
);

if (isAdmin) {
  return <AdminShell eyebrow="ADMIN" title="예약 현황" description="날짜별 전체 예약을 조회하고 승인합니다.">{innerContent}</AdminShell>;
}
return <CustomerShell title="예약 현황" description="내 예약 현황을 날짜별로 확인합니다." ...>{innerContent}</CustomerShell>;
```

### 핵심 교훈
`const Comp = conditionA ? CompA : CompB`로 동적 컴포넌트를 만들면 props가 union의 교집합(intersection)만 허용됨.  
props 구조가 다른 컴포넌트는 조건부 분기로 나눠 렌더링하는 게 안전.

## 이슈 3 — 배포 후 로그인 화면 대문 이미지 소실

### 증상
운영 사이트 로그인 페이지 좌측 이미지 영역에 alt 텍스트("대문 이미지")와 깨진 아이콘만 표시.

### 원인 파악 과정
1. DB의 `heroImageUrl` 확인 → S3 URL이 저장되어 있음
   ```
   https://beauty-book-hair-front.s3.ap-northeast-2.amazonaws.com/beauty-book/site/744f580e-...png
   ```
2. URL 직접 curl → **403**
3. `aws s3 ls s3://beauty-book-hair-front/beauty-book/` → **경로 없음**
4. `aws s3api head-object` → **404**

### 원인
프론트 배포 스크립트의 `--delete` 플래그가 S3 버킷에서 `out/` 폴더에 없는 파일을 모두 삭제함.  
백엔드를 통해 업로드된 이미지(`beauty-book/site/...`)가 `out/`에 없으므로 배포 때마다 지워짐.

```bash
# 기존 (문제)
aws s3 sync out/ s3://beauty-book-hair-front --delete

# 수정
aws s3 sync out/ s3://beauty-book-hair-front --delete --exclude "beauty-book/*"
```

### 즉각 조치
- 관리자 페이지 → 메뉴 관리 → 대문 이미지 재업로드

### 재발 방지
- 배포 가이드 `프론트엔드 배포.md`에 `--exclude "beauty-book/*"` 옵션 추가 완료
- S3 버킷 구조 원칙: 정적 파일(`_next/`, 각 페이지)과 업로드 파일(`beauty-book/`)은 prefix로 구분하여 `--delete` 범위에서 업로드 prefix를 제외

### 핵심 교훈
`aws s3 sync --delete`는 소스에 없는 파일을 대상에서 삭제함.  
백엔드가 업로드한 파일이 같은 S3 버킷에 있다면 **반드시 `--exclude`로 업로드 prefix를 보호**해야 함.  
버킷을 역할별로 분리(정적 호스팅 버킷 / 업로드 버킷)하면 이 문제가 근본적으로 없어짐.
