# 시드 데이터 자동 재생성 — 개발용 점심 차단

**날짜**: 2026-04-27
**증상**: 어드민에서 차단 시간을 삭제해도 백엔드 재시작 시 동일 데이터가 다시 생성됨. 예약 페이지에 "🍱 점심 시간 · 개발용 점심 차단" 메시지가 계속 노출됨.

---

## 증상 상세

1. 어드민 → 영업시간 관리 → 예약 차단 시간 탭에서 항목 삭제
2. 잠시 후 (혹은 백엔드 재시작 후) 예약 페이지에서 동일한 차단이 다시 보임
3. 어드민 페이지에서는 "이 달에 등록된 차단 시간이 없습니다"로 비어 보이는데, 예약 페이지에서는 "🍱 점심 시간 · 개발용 점심 차단" 표시

운영자가 만든 적 없는 사유 텍스트("개발용 점심 차단")가 표시돼서 추적이 어려웠음.

---

## 원인

**`config/ScheduleReservationSeeder.java`의 `seedLunchBlockedTime()` 메서드가 백엔드 부팅 시마다 시드 데이터를 재생성**하고 있었음.

```java
// ScheduleReservationSeeder.java (제거 전)
seedLunchBlockedTime(staffList.get(0));

private void seedLunchBlockedTime(Staff staff) {
    LocalDate nextMonday = LocalDate.now(STORE_ZONE)
            .with(TemporalAdjusters.nextOrSame(DayOfWeek.MONDAY));
    var startAt = nextMonday.atTime(13, 0).atZone(STORE_ZONE).toInstant();
    var endAt = nextMonday.atTime(14, 0).atZone(STORE_ZONE).toInstant();

    boolean exists = blockedTimeRepository
            .findByStaffIdAndStartAtLessThanAndEndAtGreaterThan(staff.getId(), endAt, startAt)
            .stream()
            .anyMatch(bt -> bt.getBlockType() == BlockedTimeType.LUNCH);
    if (exists) return;

    blockedTimeRepository.save(BlockedTime.create(
            staff, startAt, endAt, "개발용 점심 차단", BlockedTimeType.LUNCH
    ));
}
```

- `exists` 체크는 "같은 시간대에 LUNCH 타입이 이미 있으면 스킵"이지만, 어드민에서 삭제하면 다시 생성 조건이 충족됨
- 정기 차단(`recurring_blocked_times`) 기능 도입 후 이 시드는 불필요해졌으나 코드가 남아 있었음

### 추가 혼란 요인

로컬 백엔드(`gradlew bootRun`으로 실행 중인 JVM)가 한참 전에 부팅된 상태였고, 그 사이 `gradlew build`로 클래스 파일은 새로 만들어졌지만 **JVM은 메모리에 로드된 옛날 클래스를 계속 사용** 중이었음. 그래서 어드민 조회/삭제 동작이 일관되지 않아 보였음 (옛날 코드 버그가 함께 작용).

---

## 해결

### 1. 시드 메서드 호출과 정의 모두 제거

```java
// ScheduleReservationSeeder.java
- seedLunchBlockedTime(staffList.get(0));
```

`seedLunchBlockedTime` 메서드 본문도 삭제. 점심 시간은 어드민의 정기 차단(`/schedule` 영업시간 탭)에서 관리.

### 2. DB의 잔존 데이터 제거

```bash
docker exec beauty-book-postgres psql -U postgres -d beauty_book \
  -c "DELETE FROM blocked_times WHERE reason = '개발용 점심 차단';"
```

### 3. 로컬 백엔드 강제 재시작

옛날 클래스를 쥐고 있는 JVM을 종료 후 새로 띄움:

```bash
lsof -ti:4101 | xargs kill -9 || true
cd beauty-book-server && ./gradlew bootRun
```

## 교훈

- **시드 데이터에 "개발용", "테스트" 같은 식별 가능한 텍스트를 남기자** — 운영자가 만든 데이터인지 시드인지 구분 가능. 이번 케이스에선 "개발용 점심 차단"이라는 사유 덕분에 추적이 가능했음.
- **새 도메인 기능 추가 시(정기 차단) 기존 시드와의 충돌/중복을 점검하자.** 정기 차단으로 점심 시간을 관리하면서도 옛 시드가 일회성 차단을 계속 만들고 있었음.
- **`gradlew bootRun`으로 띄운 JVM은 클래스 파일이 갱신돼도 자동으로 다시 로드되지 않는다.** 코드 변경 후에는 JVM 자체를 재시작해야 함. (`spring-boot-devtools` 미적용 환경)
- **어드민 조회 결과와 고객 화면 결과가 다르면 데이터 자체가 두 군데에 있거나 캐시/조회 범위가 어긋난 것**을 의심. 이번엔 일회성 + 정기 차단이 동시에 같은 시간대를 차단했고, 정기 차단 사유가 아닌 일회성 사유가 먼저 매칭돼 표시됐음.
