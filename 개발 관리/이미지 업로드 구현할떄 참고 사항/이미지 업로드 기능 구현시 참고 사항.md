# 이미지 업로드 기능 구현 시 참고 사항

> 이 프로젝트에는 **S3 Presigned URL 기반 이미지 업로드 공통 기능**이 백엔드/프론트엔드 모두에 구축되어 있다.
> 새 기능에 이미지 업로드를 붙일 때 **백엔드는 손댈 일이 거의 없고, 프론트는 공통 함수 한 줄만 호출**하면 된다.
> 이 문서는 그 공통 자산의 위치와 사용법을 정리한다.

---

## 1. 이미지 업로드 동작 흐름 (공통)

```
[프론트] 파일 선택
    │
    ▼
[프론트] POST /api/upload/presign  ──→  [백엔드] presigned URL + publicUrl 반환
    │
    ▼
[프론트] PUT {presignedUrl}  ──→  [S3] 파일 직접 업로드
    │
    ▼
[프론트] publicUrl을 DB 저장용 데이터(글 본문/엔티티 필드 등)에 사용
```

- 파일 바이너리는 **백엔드를 거치지 않고** 프론트에서 S3로 직접 PUT 한다.
- 백엔드가 하는 일은 "어디에 어떤 키로 올릴지 정해서 서명된 URL을 발급"하는 것까지.

---

## 2. 이미지 업로드 공통 기능 (백엔드) 파일 구조

위치: `beauty-book-server/src/main/java/com/cj/beautybook/common/upload/`

| 파일 | 역할 |
|------|------|
| `UploadController.java` | `POST /api/upload/presign` 엔드포인트. ADMIN 권한 필요. MIME 타입 화이트리스트 검증 |
| `UploadService.java` | 파일명 sanitize, S3 오브젝트 키 생성, presigned URL/publicUrl 생성 |
| `S3Config.java` | `S3Client`, `S3Presigner` Bean 등록 |
| `S3Properties.java` | `aws.s3.*` 설정 바인딩 (`@ConfigurationProperties`) |

관련 설정:

- `application.yaml` `aws.s3.*` (access-key / secret-key / bucket / region / prefix / presign-expires-seconds)
- `ErrorCode.UPLOAD_NOT_CONFIGURED`, `ErrorCode.UPLOAD_INVALID_CONTENT_TYPE`

### 2-1. 엔드포인트 스펙

**Request**: `POST /api/upload/presign` (ADMIN 권한)
```json
{
  "filename": "my-photo.png",
  "contentType": "image/png",
  "folder": "board"
}
```

**Response**:
```json
{
  "presignedUrl": "https://...s3.../?X-Amz-Signature=...",
  "publicUrl":    "https://{bucket}.s3.{region}.amazonaws.com/beauty-book/board/{uuid}-my-photo.png",
  "objectKey":    "beauty-book/board/{uuid}-my-photo.png"
}
```

**제약**:
- 허용 MIME: `image/png`, `image/jpeg`, `image/jpg`, `image/webp`, `image/gif`
- presigned URL 만료: **300초(5분)**
- 파일명 sanitize: 특수문자 제거 + `{uuid}-` 접두어 자동 부여
- S3 오브젝트 키 패턴: `{prefix}/{folder}/{sanitized_filename}` (기본 prefix=`beauty-book`, folder 미지정 시 `misc`)

## 4. 위 공통 기능을 이용해 이미지 업로드 구현하는 방법

### 패턴 A. 단순 단일/다중 이미지 업로드 (가장 흔한 케이스)

> 예: 프로필 사진, 시술 서비스 대표 이미지, 사이트 설정 대문 이미지 등

```tsx
import { uploadImage } from "@/shared/api/upload";

// 단일 업로드
const url = await uploadImage(file, "your-folder-name");
// → publicUrl이 반환됨. 이 URL을 폼 state에 저장하고, 폼 submit 시 백엔드 DTO에 담아 전송

// 다중 업로드 (예: 한 번에 여러 장)
const urls = await Promise.all(
  files.map((f) => uploadImage(f, "your-folder-name"))
);
```

체크리스트:

- [ ] `<input type="file" accept="image/*" />`를 숨기고 버튼 클릭으로 트리거
- [ ] 업로드 전 클라이언트에서 파일 크기 제한 검증 (관례: **5MB**)
- [ ] 업로드 중 로딩 스피너/disabled 처리
- [ ] 업로드 성공 시 받은 `url`을 미리보기로 보여주고 폼 state에 저장
- [ ] 삭제 버튼은 폼 state에서 URL을 제거하는 것으로 충분 (S3 객체 삭제는 별도 정책)
- [ ] `folder` 이름은 **소문자 + 하이픈/언더스코어**만 사용 (예: `board`, `site`, `beauty-services`)

레퍼런스 구현:

- 단일: `features/site-settings/SiteSettingsForm.tsx` — `uploadImage(file, "site")`
- 다중: `features/beauty-service-management/BeautyServiceFormDialog.tsx` — `uploadImage(file, "beauty-services")`

### 패턴 B. 리치텍스트(Lexical) 에디터 안에서의 이미지 업로드

> 예: 게시판 글쓰기, 공지, 상담 기록 등 본문에 이미지가 섞이는 케이스

이미 `LexicalEditor`에 통합되어 있어 **에디터를 import해서 쓰기만 하면** 다음이 자동으로 동작한다:

- 툴바의 "이미지" 버튼 클릭 업로드
- 파일 드래그앤드롭 업로드
- 클립보드에서 이미지 붙여넣기 업로드
- 이미지 선택 후 리사이즈(우측 하단 핸들, `Shift`로 비율 무시)
- 좌/중/우 정렬

별도로 호출할 함수는 없다. 게시판 외의 도메인에서 같은 에디터를 쓰면 업로드 폴더가 `board`로 고정되는 점만 유의 — 도메인별 폴더로 나누고 싶다면 `shared/ui/lexical/utils/upload-image.ts`를 참고해 도메인용 래퍼를 하나 더 만들어 `LexicalEditor`에 주입하면 된다.

### 패턴 C. 백엔드 DTO 설계

이미지 URL은 **String 또는 List\<String\>** 으로 받는다. 바이너리/Multipart 받지 않는다.

```java
public record YourCreateRequest(
    String title,
    String content,
    String thumbnailUrl,         // 단일
    List<String> imageUrls       // 다중
) {}
```

엔티티 저장 시:

- 단일: `String column`
- 다중: `@ElementCollection` 또는 별도 자식 엔티티 (URL + sortOrder)

## 5. 운영 시 반드시 지켜야 할 것

### 5-1. 프론트 배포 시 `--exclude "beauty-book/*"` 필수

S3 버킷 `beauty-book-hair-front`에는 **정적 파일과 업로드 이미지가 같이 들어 있다**.
`--delete` 옵션만 쓰면 업로드 이미지 폴더(`beauty-book/`)가 통째로 날아간다.

```bash
# ✅ 올바른 명령
aws s3 sync out/ s3://beauty-book-hair-front --delete --exclude "beauty-book/*"

# ❌ 절대 금지
aws s3 sync out/ s3://beauty-book-hair-front --delete
```

자세한 사고 회고: `개발 관리/디버깅 히스토리/2026-04-26 S3 프론트 배포 시 업로드 이미지 삭제 현상 해결.md`

### 5-2. 환경변수 (EC2 `~/.env`)

```env
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_S3_BUCKET_NAME=beauty-book-hair-front
AWS_S3_REGION=ap-northeast-2
AWS_S3_PREFIX=beauty-book
```

이 중 하나라도 비면 `/api/upload/presign` 호출 시 `UPLOAD_NOT_CONFIGURED` 에러가 떨어진다.

### 5-3. CloudFront 캐시

업로드 이미지는 새 키(UUID 접두어) 기준이라 일반적으로 캐시 무효화가 필요 없다.
다만 같은 키로 덮어쓰는 시나리오를 만들면 무효화가 필요해지므로, **항상 새 키를 쓰도록 설계**한다 (`UploadService.sanitize`가 자동 처리).

## 6. 빠른 체크리스트 — 새 기능에 이미지 업로드 붙이기

- [ ] 어떤 폴더 이름을 쓸지 결정 (`board`, `site`, `beauty-services` … → 새 도메인은 새 폴더)
- [ ] 단순 업로드인지(패턴 A) 에디터 안인지(패턴 B) 결정
- [ ] 프론트에서 `uploadImage(file, folder)` 호출 → 받은 `publicUrl`을 폼 state에 저장
- [ ] 파일 크기/MIME 클라이언트 검증 (관례 5MB)
- [ ] 백엔드 DTO에 `String url` 또는 `List<String> urls` 필드 추가
- [ ] 엔티티 컬럼 추가
- [ ] 환경변수(EC2 `~/.env`) 이상 없는지 확인
- [ ] 배포 명령에 `--exclude "beauty-book/*"` 포함 확인
