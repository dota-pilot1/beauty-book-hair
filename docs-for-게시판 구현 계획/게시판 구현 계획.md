# 게시판 구현 계획

## 핵심 방향

**공통 게시판 엔진 1개 + 게시판 설정 기반 렌더링**

공지사항 / 소통 / Q&A / FAQ / 갤러리는 모두 `board_configs` 설정을 기준으로 관리한다.
게시글 저장 구조는 최대한 공통화하고, 화면 렌더링만 `kind` 값에 따라 분기한다.

핵심 원칙:

- 새 게시판 추가는 코드 수정이 아니라 `board_configs` row 추가로 처리한다.
- `boards.type ENUM`은 사용하지 않는다. enum으로 고정하면 관리자 설정으로 게시판을 추가하는 구조와 충돌한다.
- 갤러리는 별도 게시글 엔진이 아니라 `kind = GALLERY`인 게시판으로 두고, 목록 UI와 첨부 표시만 그리드 중심으로 분기한다.

---

## 게시판 종류

`board_configs.kind`는 게시판의 렌더링/운영 방식을 정하는 고정 분류다.
`board_configs.code`는 실제 게시판을 식별하는 관리자 설정값이다.

| kind | 기본 code 예시 | 설명 | 특이사항 |
|------|----------------|------|----------|
| `NOTICE` | `notice` | 공지사항 | 상단 고정 핀 기능 |
| `COMMUNITY` | `community` | 소통 게시판 | 고객 작성 허용, 댓글 |
| `QA` | `qa` | Q&A | 비공개 글, 관리자 답변 |
| `FAQ` | `faq` | 자주 묻는 질문 | 아코디언 UI, 정렬/카테고리 중요 |
| `GALLERY` | `gallery` | 이미지 갤러리 | 그리드 목록, 다중 이미지 첨부 |

예시:

- `code = notice`, `kind = NOTICE`
- `code = event`, `kind = NOTICE`
- `code = faq`, `kind = FAQ`
- `code = salon-gallery`, `kind = GALLERY`

## 백엔드 설계

### 게시판 설정 테이블

```sql
board_configs
  id,
  code,                 -- URL/API 식별자. 예: notice, community, faq
  kind,                 -- NOTICE / COMMUNITY / QA / FAQ / GALLERY
  display_name,
  description,
  allow_customer_write,
  allow_comment,
  require_admin_approval,
  is_active,
  sort_order,
  created_at,
  updated_at
```

권장 제약:

- `code` unique
- `kind`는 enum 가능. 게시판의 UI/동작 분류라서 코드 분기가 필요하다.
- 게시판 추가/비활성화/순서 변경은 이 테이블로 처리한다.

### 공통 게시글 테이블

```sql
boards
  id,
  board_config_id,
  title,
  content,
  author_id,
  status,              -- DRAFT / PENDING / PUBLISHED / HIDDEN
  visibility,          -- PUBLIC / PRIVATE
  is_pinned,
  pinned_order,
  is_answered,         -- QA용 캐시 필드
  view_count,
  created_at,
  updated_at,
  deleted_at
```

운영 규칙:

- 고객 작성 글은 설정에 따라 `PENDING`으로 생성하고, 관리자가 `PUBLISHED` 처리한다.
- 공지/FAQ처럼 관리자만 작성하는 게시판은 바로 `PUBLISHED` 가능.
- Q&A는 `visibility = PRIVATE`를 허용한다.
- soft delete는 기존 예약 도메인처럼 `deleted_at` 기준으로 맞춘다.

### 댓글 테이블

```sql
board_comments
  id,
  board_id,
  author_id,
  content,
  parent_id,
  is_admin_reply,
  created_at,
  updated_at,
  deleted_at
```

운영 규칙:

- 일반 댓글과 Q&A 답변을 같은 테이블에서 처리한다.
- Q&A 관리자 답변은 `is_admin_reply = true`인 댓글로 저장한다.
- 답변 등록/삭제 시 `boards.is_answered`를 갱신한다.
- 1차 구현에서는 대댓글을 막고 `parent_id`는 나중에 열어도 된다.

### 첨부 테이블

```sql
board_attachments
  id,
  board_id,
  file_url,
  file_name,
  content_type,
  sort_order,
  created_at
```

운영 규칙:

- 갤러리 전용 `gallery_images`를 따로 만들기보다 공통 첨부 테이블을 사용한다.
- 공지/소통/Q&A도 이미지 첨부가 필요할 수 있으므로 공통화하는 편이 낫다.
- `kind = GALLERY`인 게시판은 목록에서 첨부 이미지를 썸네일/그리드로 렌더링한다.
- 업로드는 기존 `/api/upload` presigned URL 흐름을 재사용한다.

## API 설계

### 공개/고객 API

```http
GET    /api/boards/configs
GET    /api/boards?code=notice&page=0&size=20
GET    /api/boards/{id}
POST   /api/boards
POST   /api/boards/{id}/comments
```

### 관리자 API

```http
GET    /api/admin/board-configs
POST   /api/admin/board-configs
PATCH  /api/admin/board-configs/{id}

GET    /api/admin/boards?code=notice&status=PENDING
PATCH  /api/admin/boards/{id}
DELETE /api/admin/boards/{id}
POST   /api/admin/boards/{id}/pin
DELETE /api/admin/boards/{id}/pin
```

주의:

- 공개 목록은 `is_active = true`, `status = PUBLISHED`, `deleted_at IS NULL`만 노출한다.
- 관리자 목록은 status 필터와 삭제/숨김 글 관리가 필요하다.
- `view_count` 증가는 상세 조회 API에서 처리하되, 관리자 조회에서는 증가시키지 않는다.

## 프론트 구조

현재 프로젝트는 관리자 화면도 `/dashboard`, `/reservations`, `/site-settings`처럼 루트 경로를 사용한다.
따라서 `/admin/boards`보다 기존 스타일에 맞춰 `/board-management`를 쓰는 편이 자연스럽다.

```
src/app/
├── boards/
│   └── [code]/
│       ├── page.tsx       # 고객/공개 목록
│       └── [id]/page.tsx  # 고객/공개 상세
│
├── board-management/
│   ├── page.tsx           # 관리자 게시판 설정 목록
│   ├── config/page.tsx    # 게시판 추가/설정
│   └── [code]/page.tsx    # 관리자 게시글 관리
```

FSD 구조:

```
src/entities/board/
├── api/boardApi.ts
├── model/types.ts
├── model/useBoards.ts

src/features/board-management/
├── BoardConfigFormDialog.tsx
├── BoardConfigTable.tsx
├── BoardPostTable.tsx
├── BoardPostEditor.tsx

src/widgets/board/
├── BoardList.tsx
├── BoardDetail.tsx
├── FaqAccordion.tsx
├── GalleryGrid.tsx
```

## UI 분기 기준

공통 목록/상세 컴포넌트를 기본으로 두고, `config.kind`에 따라 일부만 분기한다.

| kind | 목록 UI | 상세 UI | 작성/관리 |
|------|---------|---------|-----------|
| `NOTICE` | 리스트 | 본문 + 첨부 | 관리자 작성, 고정핀 |
| `COMMUNITY` | 리스트 | 본문 + 댓글 | 고객 작성 가능 |
| `QA` | 리스트 | 본문 + 관리자 답변 | 비공개, 답변 상태 |
| `FAQ` | 아코디언 | 필요 시 상세 생략 가능 | 관리자 정렬/카테고리 |
| `GALLERY` | 이미지 그리드 | 이미지 lightbox + 본문 | 다중 이미지 첨부 |

## 구현 우선순위

### Phase 1 — 공통 게시판 엔진

1. `board_configs`, `boards`, `board_attachments` 마이그레이션
2. `BoardConfig`, `Board`, `BoardAttachment` 엔티티/리포지토리
3. 공개 목록/상세 API
4. 관리자 게시글 CRUD API
5. 프론트 공통 목록/상세 UI
6. 관리자 게시글 관리 페이지

### Phase 2 — 운영 기능

7. 게시판 설정 관리 UI
8. 고정핀 기능
9. 고객 작성 권한과 승인 상태
10. 공통 첨부 업로드 연동

### Phase 3 — 타입별 특화

11. FAQ 아코디언 UI + 정렬/카테고리
12. Q&A 비공개 글 + 관리자 답변
13. 댓글 기능
14. 갤러리 그리드 + lightbox

## 관리자 사이드바 추가 위치

`화면 관리` 아래에 **게시판 관리** 항목 추가:

- label: `게시판 관리`
- href: `/board-management`
- icon: `MessagesSquare` 또는 `Newspaper`

공개 메뉴는 DB 메뉴 관리에 다음 형태로 추가:

- `/boards/notice` 공지사항
- `/boards/community` 소통 게시판
- `/boards/qa` Q&A
- `/boards/faq` FAQ
- `/boards/gallery` 갤러리

---

## 결론

- 공통 엔진 방향은 맞지만 `boards.type ENUM`은 제거한다.
- 게시판 추가 가능성을 살리려면 `board_configs.code`를 URL/API 기준으로 삼는다.
- 갤러리는 별도 테이블보다 공통 첨부 + `kind = GALLERY` 렌더링 분기가 낫다.
- Q&A/FAQ/승인/비공개/첨부까지 고려하면 처음부터 `status`, `visibility`, `board_attachments`를 넣는 편이 안전하다.
