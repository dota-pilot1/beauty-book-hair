# 스텝 1: DB 설계 — `menus` 테이블

> N차 트리 저장 방식은 여러 개(Adjacency List / nested set / closure table / materialized path)가 있지만, **가장 단순한 Adjacency List (`parent_id` 자기참조)** 를 선택했다.
> 메뉴 depth 는 현실적으로 3~4를 넘지 않고, 프론트가 한 번의 `SELECT *` 로 전체를 받아 트리화할 거라 이 방식으로 충분하다.
> 이 문서는 "계획" 이 아니라 **현재 엔티티 기준으로 실제 반영된 스키마 설명** 이다. 처음 구상과 달라진 부분은 아래 "결정 로그" 에 남긴다.

---

## 테이블: `menus` (Hibernate 자동 생성, 실제)

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | |
| `code` | VARCHAR(100) | UNIQUE, NOT NULL | 예) `DASHBOARD`, `ADMIN_USERS` |
| `parent_id` | BIGINT | FK → menus.id, NULL | NULL = 루트 |
| `label` | VARCHAR(100) | NOT NULL | 폴백용 라벨 |
| `label_key` | VARCHAR(100) | NULL | i18n 키 예) `nav.users` |
| `path` | VARCHAR(255) | NULL | 내부 경로 (폴더형이면 NULL) |
| `icon` | VARCHAR(100) | NULL | lucide 아이콘명 |
| `is_external` | BOOLEAN | NOT NULL, default false | true 면 외부 URL |
| `required_role` | VARCHAR(100) | NULL | 예) `ROLE_ADMIN` |
| `required_permission` | VARCHAR(100) | NULL | (현재 미사용) |
| `visible` | BOOLEAN | NOT NULL, default true | soft-hide |
| `display_order` | INT | NOT NULL, default 0 | 같은 parent 안에서 정렬 |
| `created_at` | TIMESTAMP | NOT NULL | Hibernate `@CreationTimestamp` |
| `updated_at` | TIMESTAMP | NOT NULL | Hibernate `@UpdateTimestamp` |

> 스키마는 엔티티([02-도메인-모델-JPA.md](./02-도메인-모델-JPA.md))에서만 관리하고 `ddl-auto: update` 로 자동 동기화. 수동 마이그레이션 SQL 없음.

## 필드 설계 근거

| 필드 | 근거 |
|------|------|
| `code` UNIQUE | 시더/링크 타겟의 안정적 키. `id` 는 재시드 시 흔들림. |
| `parent_id` self-FK | Adjacency List. NULL = 1차 메뉴. |
| `label_key` + `label` | `label_key` 있으면 `t(label_key)`, 없으면 `label` 폴백. |
| `path` nullable | "관리" 같은 폴더형 메뉴(클릭해도 이동 안 함)는 NULL. |
| `is_external` | 외부 링크면 `target="_blank"`. |
| `required_role` / `required_permission` | 둘 다 NULL = 로그인 유저 누구나. **현재는 `required_role` 만 활용.** |
| `visible` | 삭제 대신 숨김 기능 (Hibernate 자동 삭제 방지). |
| `display_order` | 같은 `parent_id` 내에서만 의미. 전역 순서 아님. 현재 시더는 **연속값(0,1,2,...)**, DnD 저장도 연속값. (10단위 띄우기 미적용) |

## 인덱스 & 제약 (실제)

현재 엔티티에서 명시한 핵심 제약은 **`code` 유니크 + `parent_id` FK** 다.

```
UNIQUE  uk_menus_code (code)
FK      parent_id → menus(id)    -- Hibernate 기본 FK (ON DELETE 제약 없음)
```

- `@Table(uniqueConstraints = ...)` 와 `@Column(unique = true)` 를 둘 다 걸었지만, 의도는 동일하게 **`code` 유니크 보장** 하나다.
- 원래 계획엔 `INDEX(parent_id, display_order)` + `ON DELETE RESTRICT` 를 두려 했지만, 실제 메뉴 row 수가 수십 건 수준이라 생략했다.
- 현재 `MenuService.delete()` 는 **자식 존재 선검증 없이 바로 delete** 한다. 자식 있는 row 삭제는 DB FK 에서 막히고, 관리자 UI 는 그 전에 confirm 만 한 번 띄우는 수준이다.
- 자식 존재 여부를 사용자 친화적으로 안내하려면 추후 `MenuService.delete()` 에 선제 검증을 넣는 편이 맞다.

## Depth 를 저장하지 않는 이유

- `depth` 는 `parent_id` 체인 따라 계산 가능한 **파생값**.
- 저장하면 부모 변경 시 자식 전체 depth 재계산 필요 → 일관성 비용 ↑.
- 프론트 `buildTree` 가 재귀적으로 자연 처리. 현재 구현은 `depth` 필드도 아예 쓰지 않고 트리의 구조 자체로 들여쓰기를 렌더.

## 엣지 케이스 정책 (현재)

| 케이스 | 정책 | 구현 상태 |
|--------|------|:--------:|
| `parent_id = 자기 자신` | 애플리케이션 레벨에서 차단해야 함 | ❌ 미구현 (UI 에서 루트만 선택 가능해 현재는 발생 불가) |
| 순환 참조 (A→B→A) | 부모 변경 시 조상 체인 체크 | ❌ 미구현 (동일 이유) |
| `display_order` 충돌 | 정렬 `(display_order ASC, id ASC)` 로 안정화 | ⚠️ 1차 정렬만 구현, 2차는 Map 순회 순서에 의존 |
| 폴더형인데 자식 전부 숨김 | 빈 드롭다운 방지 | ❌ 미구현 (현재 시더엔 발생 케이스 없음) |

> 부모 자유 선택(다른 부모로 이동)이 UI 에 추가되는 시점에 위 3가지를 반드시 구현해야 한다.

## 결정 로그

| 주제 | 원래 계획 | 실제 구현 | 이유 |
|------|----------|-----------|------|
| 트리 저장 방식 | closure table 등 확장 고려 | `parent_id` self-FK | 메뉴 수 적고 조회가 단순 |
| 인덱스 | `INDEX(parent_id, display_order)` 추가 | 미적용 | 현재 row 수가 적어 체감 이점 낮음 |
| 삭제 제어 | DB/서비스 둘 다 자식 삭제 방어 | DB FK 에만 실질 의존 | 구현 복잡도보다 단순함 우선 |
| depth 컬럼 | 저장 가능성 열어둠 | 미저장 | 파생값이라 동기화 비용만 증가 |

## DDL 확인 방법

`ddl-auto: update` 로 실행 후:

```sql
SHOW CREATE TABLE menus;
```

또는 `beauty-book-server/build/` 생성 후 실제 MySQL 확인. 스키마 변경 시 엔티티만 커밋하고 DB 는 앱 기동으로 동기화.
