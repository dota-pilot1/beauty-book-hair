# 페이지 구조 설계

## 현재 판단

- 지금 단계에서는 메인에 예약 기능을 바로 크게 올리기보다, 역할별 첫 진입 화면을 먼저 분리하는 방향이 맞음.
- 이유: 현재 프로젝트는 인증/권한/관리 메뉴 기반이 먼저 갖춰져 있어서 관리자 예약 관리 MVP를 먼저 붙이기 좋음.
- 고객용 예약 메인은 예약 도메인, 예약 가능 시간 계산, 직원/서비스 데이터가 준비된 뒤 확장하는 편이 안전함.

## 큰 방향

### 사용자 유형 분리

1. 관리자/매장 운영자
2. 고객

### 로그인 후 첫 화면 원칙

- 관리자/매장 운영자에게는 운영 대시보드를 보여줌
- 고객에게는 예약 중심 홈 또는 내 예약 중심 화면을 보여줌

## 추천 라우팅 정책

### 공통

- `/` : 비로그인 진입 시 소개 또는 로그인 유도
- `/login` : 로그인
- `/register` : 회원가입

### 관리자/운영자

- `ROLE_ADMIN`, `ROLE_MANAGER` 로그인 후 기본 이동: `/dashboard`
- `/dashboard` 는 빈 랜딩이 아니라 예약 운영 대시보드로 개편

### 고객

- `ROLE_USER` 로그인 후 기본 이동: `/booking` 또는 `/reservations`
- 초반에는 `/booking` 이 더 직관적

## 왜 이렇게 가는지

### 관리자 관점

- 로그인 직후 가장 필요한 것은 "오늘 운영 상황 확인"임
- 따라서 첫 화면은 예약 현황, 곧 시작할 예약, 빠른 관리 진입 버튼이 있어야 함

### 고객 관점

- 로그인 직후 가장 필요한 것은 "예약하기" 또는 "내 예약 보기"임
- 따라서 고객 첫 화면은 예약 플로우 중심이어야 함

## 1차 구현 추천 구조

### 1. 관리자용 먼저

- `/dashboard`
- `/reservations`
- `/customers`
- `/staff`
- `/services`

### 2. 고객용은 그 다음

- `/booking`
- `/my-reservations`
- `/profile`

## 로그인 후 보여줄 화면 추천

### 관리자 대시보드

- 오늘 예약 수
- 곧 시작할 예약
- 취소/노쇼 현황
- 직원별 오늘 스케줄 요약
- 빠른 이동
- `예약 관리`
- `고객 관리`
- `직원 관리`

### 고객 예약 홈

- 예약하기 CTA
- 시술/서비스 선택
- 디자이너 선택
- 가능한 시간 보기
- 내 예약 확인

## 현재 단계에서의 우선순위 판단

지금 바로 해야 할 것은 아래 순서가 적절함.

1. 역할별 로그인 후 리다이렉트 정책 확정
2. 관리자용 예약 관리 페이지 설계
3. 관리자용 대시보드 내용을 예약 운영 중심으로 개편
4. 예약 도메인 API 및 테이블 설계
5. 이후 고객용 예약 메인 설계

## 추천 메뉴 구조

### 관리자 메뉴

- 대시보드
- 예약 관리
- 고객 관리
- 직원 관리
- 서비스 관리
- 메뉴 관리
- 사이트 설정

### 고객 메뉴

- 예약하기
- 내 예약
- 프로필

## 결론

- 지금은 "메인에 예약을 바로 크게 노출"하는 것보다 "로그인 후 역할별로 다른 첫 화면"을 두는 것이 맞음.
- 1차 구현은 관리자용 예약 관리와 운영 대시보드를 먼저 만드는 방향이 가장 자연스러움.
- 고객용 예약 메인은 예약 데이터 구조와 운영 규칙이 정리된 뒤 붙이는 것이 좋음.
